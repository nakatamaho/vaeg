## FDD slice evidence and eventual convergence

The existing slice-exact reproducer remains unchanged and deliberately fails
comparison. With bytes `18 02 00 00 d3 f4`, A=`0x5a`, and shared slices
`1,7`, legacy's 7-clock taken JR executes `OUT (0xf4),A` in the second
`Exec()`. The new core's architectural 12-clock JR leaves PC=`0x0004` and no
I/O event in that slice. The first difference remains
`cycle-fdd-io-scheduling/fdd-io-slice/0`:

```text
legacy: pc=0006 r=02 events=read 0004:d3, read 0005:f4, out f4:5a
new:    pc=0004 r=01 events=none
```

Port `0xf4` calls `fdcsubsys_o_dskctl`, so this is not relabelled as
externally inert and the reproducer is not allowlisted or removed.

The green `cycle-fdd-io-eventual-convergence` case uses the same initial
program and `1,7` slices, then gives both runners identical additional `4,1`
slices. At 13 total clocks, both aggregated traces are exactly:

```text
read 0000:18, read 0001:02, read 0004:d3, read 0005:f4, out f4:5a
```

The comparator separately requires exactly one I/O event and exactly one
`OUT (0xf4),0x5a` in each trace. Both finish with PC=`0006`, R=`02`,
AF=`5a00`, SP=`f000`, identical remaining architectural and interrupt state,
memory hash `1b7d5566a78bcbd6`, device hash `9fe556f5e2283885`, and
`lastclock=13`. No event is lost, duplicated, or reordered. The classified
remaining differences are consumed clocks `18/23`, `remainclock=-5/-10`,
and the `Exec()` slice containing the I/O event.

G38 blocks on lost, duplicated, reordered, or permanently divergent external
effects. A slice shift caused by a verified architectural timing correction
may pass when eventual convergence is proven. This difference remains a
mandatory M39 private integration risk. M39 must stop production integration
if real VA/FDD testing finds transfer failure, timeout, corrupted data,
changed IRQ/DRQ sequencing, or broken save/load behavior. M38 adds no legacy
per-opcode timing emulation.

## M39 production-seam status

M39 preserves this risk and adds no legacy cycle emulation. A ROM-less
production-subsystem test emits exactly one `OUT (0xf4),0x5a` through
`fdcsubsys_o_dskctl`, saves after the CPU call returns, reloads, and resumes
without losing or duplicating that output under either production selection.
This proves the adapter and frame-boundary plumbing, not private FDD timing.

The M39 private manifest subsequently passed under both core selections. A
timing-sensitive loader reached the same stable result after a later
intermediate wall-clock sample under the new core. Both traces contained the
same 21 ordered port-`0xf4` values; their first 164 completed FDC records were
byte-identical, and the longer legacy capture contained only additional
post-convergence records. Read/write, loader, SLEEP_HACK/WAIT, legacy-to-new,
new-to-new, and active-loader state cases had no timeout, corruption, lost or
duplicated event, failed wake, or changed completion.

This evidence discharges the mandatory G39 integration risk without changing
the M38 classification or adding legacy cycle emulation. The current private
trace has no dedicated DRQ-edge record; FDC DMA transfer metadata, IRQ and
acknowledge events, final data, and guest-visible completion were checked.
